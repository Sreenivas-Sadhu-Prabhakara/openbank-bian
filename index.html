What is what

Each capability of the bank is its own service: it owns its data, speaks Open Banking on the wire, and is deployed and scaled on its own. One service — Consent — sits at the centre; nothing is read or paid without an authorised consent it has issued.

02

Why this matters

Banking software has spent decades as a single, indivisible core — hard to change, harder to open. Open Banking regulation forced the doors open; this estate shows the architecture that makes the openness safe, standard and sustainable.

01

Standards instead of bespoke integrations

Every service speaks the same OBIE contract on the wire. A fintech integrates the standard once, not a different proprietary API per bank. The cost of connecting the financial system collapses from N×M to N+M.

OBIE Read/Write APIs
02

Consent-first, by construction

No service trusts the caller's claim of access. Each one fetches and checks the consent — its status, type, permissions and expiry — from the Consent service before doing anything. The customer's authorisation is a hard gate in code, not a policy in a document.

authorise → validate → consume → revoke
03

Capabilities as bounded domains

Following BIAN, each business capability is an independently ownable service domain with its own data and its own schema. Teams ship Accounts without touching Payments. The org chart and the architecture finally agree.

BIAN service domains
04

Replaceable parts, explicit contracts

Behind each service is a clean port: an in-memory store for tests, Postgres for real, a fake client for the upstream. You can swap the datastore, restub a dependency, or rewrite a service entirely — as long as the contract holds.

hexagonal · ports & adapters
05

A reference you can run, not just read

It builds, it tests, it boots with zero infrastructure. Turning an abstract standard into something a team can go run in a minute is how good architecture actually spreads.

in-memory dev · Docker estate

03

How it helps banking evolve

The same five-service shape points at where the industry is going: away from the monolithic core, toward an ecosystem where banking is embedded, governed by consent, and increasingly acted on by software agents.

01

From a frozen core to a composable bank

When capabilities are independent services, the bank evolves the parts instead of risking the whole. New products become a recombination of existing domains rather than another change request against a decades-old core that no one dares touch.

02

Open ecosystems & embedded finance

Standard, consent-gated APIs let fintechs, marketplaces and non-banks build on the bank's rails. Payments and account insight show up inside the products people already use — and the bank earns reach it could never build alone.

03

Compliance as a feature, not a bolt-on

The consent lifecycle — granted, scoped, time-boxed, revocable, fully audited — is modelled in the code itself. Regulators get provable control; customers get a clear switch; the bank gets trust it can demonstrate, not just assert.

04

A foundation for agentic finance

As AI agents begin to move money and read accounts on a customer's behalf, the consent boundary becomes essential infrastructure: an agent operates only within an explicit, permissioned, revocable grant. The same gate that protects a human is what keeps autonomous software safe.

05

A lower cost of change

Swap Postgres for another store, add a second market's API flavour (Berlin Group, FDX), split a hot domain or scale it alone — each is a local move behind a stable contract, not a bank-wide migration. Evolution becomes routine instead of existential.

04

The repositories

One repository per microservice. Each is fully self-contained — its own SQL migrations, a vendored copy of the shared library, a Dockerfile and tests — so it clones and builds on its own.
